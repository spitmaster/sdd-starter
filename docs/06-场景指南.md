# SDD 开发场景指南

> Specification-Driven Development 场景化使用手册

---

## 三种开发场景

| 场景 | 你的情况 | 推荐模式 |
|------|---------|----------|
| A | 有完整 PRD、设计稿、需求文档 | **PRD 导入模式** |
| B | 只有模糊想法，边做边看 | **敏捷探索模式** |
| C | 已有项目，继续开发 | **项目接续模式** |

---

## 场景 A：PRD 导入模式

### 适用情况
- 已有产品需求文档（PRD）
- 有设计稿或 UI 规范
- 需求明确，需要严格执行

### 工作流程

```
1. 你提供 PRD 文档（粘贴内容或文件路径）
       │
       ▼
2. AI 分析 PRD，提取关键信息
       │
       ▼
3. AI 询问遗漏细节（如有）
       │
       ▼
4. AI 生成完整 SPEC.md
       │
       ▼
5. 你确认 SPEC（可以修改）
       │
       ▼
6. AI 严格按 SPEC 开发
       │
       ▼
7. 开发完成，对照 SPEC 验收
```

### 使用方法

```bash
# 方式 1：粘贴 PRD 内容
> 我有一个 PRD，需求如下：[粘贴 PRD 内容]
> 请帮我分析并生成 SPEC.md

# 方式 2：提供文件路径
> 请读取 docs/PRD.md，然后基于它生成 SPEC.md
```

### AI 提示词模板

```
请作为 SDD 架构师，分析以下 PRD 并生成 SPEC.md：

1. 首先识别项目类型和技术栈
2. 提取所有功能需求
3. 识别模糊或遗漏的地方，询问我确认
4. 补充非功能需求（性能、安全、兼容性等）
5. 生成完整 SPEC.md

PRD 内容：
[粘贴你的 PRD]
```

## 场景 B：敏捷探索模式

### 适用情况
- 想法不成熟，还在探索
- 需要看原型再决定
- 边做边改，没有固定需求

### 工作流程

```
1. 你描述一个大概的想法
       │
       ▼
2. AI 做一个最小原型（5-10 分钟）
       │
       ▼
3. 你体验原型，说出感受
       │
       ▼
4. AI 根据反馈调整/新增功能
       │
       ▼
5. 循环 2-4，直到满意
       │
       ▼
6. AI 整理 SPEC.md（逆向生成）
```

### 使用方法

```bash
# 开始探索
> 我想做个记账应用，但具体什么样还不知道，先做个简单的看看

# 体验后反馈
> 不错，但我觉得分类太少了，能不能再加个自定义分类功能

# 继续迭代
> 我想再加个统计图表功能
```

### AI 提示词模板

```
请以敏捷开发模式启动这个项目：

1. 先做一个最小可用的原型（MVP）
   - 只包含核心功能
   - 不需要完美，能演示即可
   - 完成后主动让我体验

2. 每次迭代：
   - 询问我体验感受
   - 记录我反馈的问题
   - 调整后继续开发

3. 项目完成后：
   - 整理一份 SPEC.md
   - 包含已完成的功能
   - 列出可能的扩展点

当前想法：[描述你的想法]
```

## 场景 C：项目接续模式

### 适用情况
- 已有项目代码，需要继续开发
- 可能是修 bug、加功能、重构
- 需要先了解项目再动手

### 工作流程

```
1. AI 自动扫描项目结构
       │
       ▼
2. AI 分析代码，识别项目类型
       │
       ▼
3. AI 检测是否有 SPEC.md
       │
       ├── 有 SPEC → 读取并继续开发
       │
       └── 无 SPEC → 询问并生成
       │
       ▼
4. AI 识别开发模式
       │
       ├── 修 Bug → 分析代码找问题
       ├── 加功能 → 分析现有架构
       └── 重构 → 制定重构计划
       │
       ▼
5. 执行任务
```

### 使用方法

```bash
# 启动项目接续
> 请分析当前项目，帮我确定接下来的开发计划

# 或者更具体
> 项目中有几个 bug，请帮我分析并修复

# 或者
> 请在现有项目上添加登录功能
```

### AI 提示词模板

```
请以项目诊断模式启动：

1. 自动扫描项目结构
   - 识别技术栈（语言、框架、依赖）
   - 识别项目架构（前后端分离、微服务等）
   - 检测是否有 SPEC.md

2. 分析代码质量
   - 识别代码风格
   - 检测明显问题
   - 评估代码复杂度

3. 确定开发模式
   - 如果有 SPEC → 按 SPEC 继续
   - 如果没有 → 生成 SPEC 后继续
   - 如果有 bug → 分析并修复
   - 如果要加功能 → 分析架构后实现

4. 给出项目诊断报告
   - 项目概况
   - 发现的潜在问题
   - 接下来的开发建议
```

## 项目诊断器（自动识别）

### 诊断规则

| 检测项 | 结果 | 建议模式 |
|--------|------|----------|
| 有 `SPEC.md` 且完整 | 规范项目 | 按 SPEC 继续 |
| 有 `SPEC.md` 但不完整 | 待完善项目 | 补充 SPEC |
| 无 `SPEC.md`，有大量代码 | 遗留项目 | 生成 SPEC + 分析 |
| 无代码，有 PRD 文档 | 新项目 | PRD 导入 |
| 目录为空或很少文件 | 新项目 | 敏捷探索 |
| 有 `package.json` | Node.js 项目 | 技术栈识别 |
| 有 `pom.xml` | Java 项目 | Maven 项目 |
| 有 `requirements.txt` | Python 项目 | pip 项目 |

## 快速开始命令

```bash
# 通用启动命令（在项目目录下执行）
> 请先阅读 README.md，然后诊断当前项目情况

# 新项目启动
> 请以敏捷探索模式启动项目

# PRD 导入
> 请分析 docs/prd.md 并生成 SPEC.md

# 继续开发
> 请分析当前项目并制定开发计划
```

## 选择你的模式

根据你的情况，选择合适的启动方式：

1. **有完整 PRD** → 使用 `docs/PRD导入指南.md` 或直接粘贴 PRD
2. **只有模糊想法** → 直接描述你的想法，后面交给 AI
3. **已有项目** → 在项目目录下启动，让 AI 自动诊断
