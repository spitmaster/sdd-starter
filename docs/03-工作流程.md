# SDD 工作流程

> Specification-Driven Development 标准化开发流程

---

## 流程概览

```
需求收集 → SPEC编写 → 开发实现 → 自测验证 → 验收确认
```

---

## 阶段 1: 需求收集

### 输出
- 原始需求文档
- 用户访谈记录

### 检查清单
- [ ] 明确了目标用户
- [ ] 明确了核心问题
- [ ] 明确了价值主张

---

## 阶段 2: SPEC 编写

### 输出
`SPEC.md` - 项目规格说明书

### 必填章节

1. **项目概述** - 名称、背景、目标用户
2. **功能需求** - 每个功能的描述和验收标准
3. **非功能需求** - 性能、安全、兼容性
4. **UI/UX 规范** - 页面结构、视觉规范
5. **技术架构** - 技术栈、项目结构
6. **API 设计** - 接口定义、请求响应示例
7. **验收标准** - 完成标准清单

### 检查清单
- [ ] 所有功能都有验收标准
- [ ] 验收标准都是可测试的
- [ ] UI 规范有具体数值
- [ ] API 有请求/响应示例

---

## 阶段 3: 开发实现

### 开发原则

1. **严格遵循 SPEC**
   - 不添加 SPEC 外的功能
   - 不省略 SPEC 中的要求

2. **保持同步更新**
   - SPEC 变更 → 代码同步
   - 代码实现 → SPEC 更新

### 检查清单
- [ ] 代码符合 SPEC 规格
- [ ] SPEC 已更新（如有变更）
- [ ] 单元测试已编写

---

## 阶段 4: 自测验证

### 验证步骤

1. **功能验证** - 对照 SPEC 逐项检查
2. **边界测试** - 空值、异常、并发
3. **回归测试** - 确保未破坏已有功能

### 检查清单
- [ ] 所有验收标准已测试
- [ ] 边界情况已覆盖
- [ ] 回归测试通过

---

## 阶段 5: 验收确认

### 验收流程

1. **演示功能** - 按 SPEC 演示每个功能
2. **确认文档** - 检查 SPEC 是否完整
3. **签署验收** - 记录遗留问题（如有）

---

## 变更管理

### 变更时必做

1. 更新 SPEC 相关章节
2. 注明变更原因和日期
3. 评估对其他功能的影响
4. 重新验证相关功能
