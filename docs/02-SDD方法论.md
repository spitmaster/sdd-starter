# SDD 方法论

> Specification-Driven Development 核心概念

---

## 什么是 SDD

SDD（Specification-Driven Development）是一种以规格文档为核心的開發方法论。

**核心理念**：先明确"做什么"，再动手"怎么做"。

---

## SDD vs 传统开发

| 维度 | 传统开发 | SDD |
|------|----------|-----|
| 起点 | 代码 | SPEC |
| 需求变更 | 口头通知 | 必须更新文档 |
| 验收 | 主观判断 | 客观标准 |
| 交接 | 依赖口头解释 | 文档即说明 |

---

## SDD 的核心价值

### 1. 减少返工
- 需求不明确就开始写代码 → 返工率高
- 需求明确后再写代码 → 一次做对

### 2. 提高质量
- 每个功能都有明确的验收标准
- 验收标准 = 测试用例

### 3. 便于交接
- 新人看文档就能上手
- 不依赖"问人"

---

## SDD 四步法

```
需求 ──▶ SPEC ──▶ 开发 ──▶ 验收
  │        │        │        │
  ▼        ▼        ▼        ▼
 明确    文档化   按规实现  对照检查
```

---

## 好的 SPEC 特征

| 特征 | 说明 |
|------|------|
| **完整** | 包含所有功能点 |
| **明确** | 没有歧义描述 |
| **可测试** | 每条标准都能验证 |
| **一致** | 前后描述不矛盾 |

---

## 常见误区

### ❌ 误区 1：SPEC 越详细越好
> 详细不等于繁琐。聚焦于"做什么"，不要写"怎么做"。

### ❌ 误区 2：先写代码再补 SPEC
> 倒着想容易，正着做难。SPEC 的价值在于"提前发现问题"。

### ❌ 误区 3：SPEC 不能改
> 需求变化时，SPEC 必须同步更新。变更本身不是问题，不更新才是。
