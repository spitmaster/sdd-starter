# 项目诊断器

> AI 启动时自动执行的诊断逻辑

---

## 诊断流程

```
START
  │
  ▼
检测目录结构
  │
  ├─ 有代码文件？ ──► 是 ──► 识别技术栈
  │                           │
  │                           ▼
  │                    有 SPEC.md？
  │                      │
  │              ┌───────┴───────┐
  │              │               │
  │             是              否
  │              │               │
  │              ▼               ▼
  │       按 SPEC 继续    生成/补充 SPEC
  │              │               │
  │              └───────┬───────┘
  │                      │
  │                      ▼
  │              确定开发模式
  │              (新功能/Bug/重构)
  │                      │
  │                      ▼
  └─► 否 ──► 新项目 ←── 结束
                  │
                  ▼
          敏捷探索模式
```

## 诊断检查清单

### 1. 项目结构检测

| 必检文件 | 说明 |
|---------|------|
| `package.json` | Node.js |
| `requirements.txt` | Python |
| `pom.xml` | Java |
| `go.mod` | Go |
| `Cargo.toml` | Rust |
| `composer.json` | PHP |
| `app.json` | 微信小程序 |
| `manifest.json` | uni-app |

### 2. 技术栈识别

| 特征 | 技术栈 |
|------|--------|
| `package.json` | Node.js |
| `requirements.txt` | Python |
| `pom.xml` | Java |
| `go.mod` | Go |
| `Cargo.toml` | Rust |
| `app.json` | 微信小程序 |

### 3. 项目类型识别

| 类型 | 特征 |
|------|------|
| Web 前端 | HTML/CSS/JS, React/Vue/Angular |
| Web 后端 | API 服务器, 数据库 |
| CLI 工具 | 无 UI, 命令行 |
| 桌面应用 | Electron/Tauri |
| 移动应用 | React Native/Flutter |
| 小程序 | app.json, pages/ |

### 4. SPEC 完整性检测

- [ ] 1. 项目概述
- [ ] 2. 功能需求
- [ ] 3. 非功能需求
- [ ] 4. UI/UX 规范
- [ ] 5. 技术架构
- [ ] 6. API 设计
- [ ] 7. 验收标准

## 诊断输出模板

```markdown
# 项目诊断报告

## 1. 项目概况
- **项目名称**: [从 README 或目录名获取]
- **技术栈**: [识别到的技术栈]
- **项目类型**: [Web/CLI/桌面/移动/小程序]

## 2. SPEC 状态
- **状态**: 完整/待补充/不存在
- **缺失章节**: [如有缺失，列出]

## 3. 推荐开发模式
- [ ] 新功能开发
- [ ] Bug 修复
- [ ] 重构

## 4. 建议的下一步
1. [第一步建议]
2. [第二步建议]
```

## AI 使用指南

```bash
# 启动诊断
> 请诊断当前项目并生成诊断报告

# 根据诊断结果操作
> 请帮我补充缺失的 SPEC 章节
> 请在现有架构上添加 [功能描述]
> 请分析并修复 [问题描述]
```
