# SDD 工作流程

> Specification-Driven Development 标准化开发流程

---

## 流程概览

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  需求   │───▶│  SPEC   │───▶│  开发   │───▶│  自测   │───▶│  验收   │
│  收集   │    │  编写   │    │  实现   │    │  验证   │    │  确认   │
└─────────┘    └─────────┘    └─────────┘    └─────────┘    └─────────┘
     │              │              │              │
     └──────────────┴──────────────┴──────────────┘
                        (变更时循环)
```

---

## 阶段 1: 需求收集

### 目标
将业务需求转化为可理解的输入

### 输出
- 原始需求文档
- 用户访谈记录
- 竞品分析（如有）

### 检查清单
- [ ] 明确了目标用户
- [ ] 明确了核心问题
- [ ] 明确了价值主张
- [ ] 识别了关键约束

---

## 阶段 2: SPEC 编写

### 目标
创建完整、可测试的规格文档

### 输出
`SPEC.md` - 项目规格说明书

### 步骤

#### 2.1 编写概述
```markdown
## 1. 项目概述
- 项目名称
- 项目简介（2-3句话）
- 目标用户
- 项目背景
```

#### 2.2 定义功能需求
```markdown
## 2. 功能需求

### 功能 1: [名称]
- 描述
- 用户故事
- 验收标准（每条必须可测试）
```

#### 2.3 定义非功能需求
```markdown
## 3. 非功能需求
- 性能要求
- 安全要求
- 可用性要求
```

#### 2.4 编写 UI/UX 规范
```markdown
## 4. UI/UX 规范
- 页面结构
- 视觉设计（颜色、字体、间距）
- 组件规范
```

#### 2.5 设计技术架构
```markdown
## 5. 技术架构
- 技术栈
- 项目结构
- API 设计
```

#### 2.6 定义验收标准
```markdown
## 7. 验收标准
- 完成标准清单
- 测试用例
```

### 检查清单
- [ ] 所有功能都有验收标准
- [ ] 验收标准都是可测试的
- [ ] UI 规范有具体数值
- [ ] API 有请求/响应示例
- [ ] 文档格式统一

---

## 阶段 3: 开发实现

### 目标
按照 SPEC 实现功能

### 开发原则

1. **严格遵循 SPEC**
   - 不添加 SPEC 外的功能
   - 不省略 SPEC 中的要求

2. **保持同步更新**
   - SPEC 变更 → 代码同步
   - 代码实现 → SPEC 更新

3. **频繁验证**
   - 每完成一个小功能就验证

### AI 辅助开发

使用 AI 工具时，提示词示例：

```
请严格按照 SPEC.md 中的规格实现功能：
1. 功能需求见第 2 节
2. UI 规范见第 4 节
3. 验收标准见第 7 节

每完成一个功能，告诉我是否符合 SPEC。
```

### 检查清单
- [ ] 代码符合 SPEC 规格
- [ ] SPEC 已更新（如有变更）
- [ ] 单元测试已编写
- [ ] 代码风格统一

---

## 阶段 4: 自测验证

### 目标
验证实现满足 SPEC

### 验证步骤

#### 4.1 功能验证
对照 SPEC 中的验收标准逐项检查：

```markdown
验收标准 1: [标准内容]
- 测试方法: [如何测试]
- 测试结果: [通过/不通过]
- 问题（如有）: [描述]
```

#### 4.2 边界测试
- 空值处理
- 异常输入
- 并发场景
- 性能边界

#### 4.3 回归测试
确保新功能未破坏已有功能

### 检查清单
- [ ] 所有验收标准已测试
- [ ] 边界情况已覆盖
- [ ] 回归测试通过
- [ ] 性能达标

---

## 阶段 5: 验收确认

### 目标
确认产品满足业务需求

### 验收流程

1. **演示功能**
   - 按 SPEC 演示每个功能
   - 展示边界情况处理

2. **确认文档**
   - 检查 SPEC 是否完整
   - 确认文档与实现一致

3. **签署验收**
   - 相关方签字确认
   - 记录遗留问题（如有）

### 检查清单
- [ ] 功能演示完成
- [ ] SPEC 文档已签署
- [ ] 遗留问题已记录
- [ ] 验收结论明确

## 变更管理

### 变更流程

```
发现变更需求
    │
    ▼
评估变更范围
    │
    ├── 影响小 ──▶ 直接修改 SPEC 和代码
    │
    └── 影响大 ──▶ 重新走完整流程
```

### 变更时必做

1. 更新 SPEC 相关章节
2. 注明变更原因和日期
3. 评估对其他功能的影响
4. 同步更新测试用例
5. 重新验证相关功能

## 快速参考

### 各阶段产出

| 阶段 | 主要产出 | 文件 |
|------|----------|------|
| 需求收集 | 原始需求 | requirements.md |
| SPEC 编写 | 规格文档 | SPEC.md |
| 开发实现 | 可运行代码 | src/ |
| 自测验证 | 测试报告 | tests/ |
| 验收确认 | 验收报告 | docs/reports/ |

### 关键命令

```bash
# 初始化 SDD 项目
.\scripts\init-project.ps1 -ProjectName "my-app"

# 验证 SPEC 完整性
.\scripts\validate-spec.ps1

# 检查项目结构
.\scripts\validate-spec.ps1 -CheckStructure
```
