# SDD 最佳实践

> 在实际项目中应用 SDD 的经验总结

---

## SPEC 编写最佳实践

### 编写技巧

#### 1. 使用精确的语言

❌ 模糊表述：
> "界面要美观"  
> "系统要快"  
> "用户应该能方便地..."

✅ 精确表述：
> "主色调为 #1890FF，圆角为 8px"  
> "API 响应时间 < 200ms，P95 < 500ms"  
> "用户可通过点击按钮导出报表，文件格式为 CSV"

#### 2. 验收标准格式

每个验收标准采用以下格式：

```markdown
**Given** [前置条件]
**When** [操作]
**Then** [预期结果]
```

示例：
```markdown
**Given** 用户已登录且有管理员权限
**When** 用户访问 /admin/users
**Then** 显示用户列表，最多显示 100 条
**And** 每条显示：用户名、邮箱、角色、注册时间
**And** 支持分页，每页 20 条
```

#### 3. 数值化描述

| 方面 | 正确写法 |
|------|----------|
| 时间 | "响应时间 < 200ms" |
| 数量 | "最多支持 1000 并发" |
| 尺寸 | "图片最大 2MB" |
| 百分比 | "成功率 > 99.9%" |
| 位置 | "按钮距左侧 16px，距顶部 24px" |

## 项目结构最佳实践

### 目录组织

```
project/
├── README.md              # 项目入口（必须）
├── SPEC.md                # 规格文档（必须）
├── CHANGELOG.md           # 变更日志
├── docs/                  # 详细文档
│   ├── api/              # API 文档
│   ├── guides/           # 使用指南
│   └── reports/          # 测试报告
├── src/                   # 源代码
├── tests/                 # 测试代码
├── scripts/               # 工具脚本
├── configs/               # 配置文件
└── .sdd/                  # SDD 配置
    ├── config.json
    └── prompts/
```

### 文件命名

- 使用小写字母 + 连字符
- README.md（大写固定）
- CHANGELOG.md（大写固定）
- 其他：kebab-case

## AI 协作最佳实践

### 提示词优化

#### 1. 结构化提示词

```markdown
## 任务
[具体任务描述]

## 参考文档
- SPEC.md 第 2 节
- docs/api.md

## 约束
- 严格遵循 SPEC 规格
- 不添加未定义的功能
- 代码风格见 .eslintrc.js

## 输出要求
- 先解释思路
- 再提供代码
- 最后说明验收标准检查结果
```

#### 2. 分步骤执行

不要一次性给 AI 太多任务：

```markdown
第一步：阅读 SPEC.md，确认登录功能规格
第二步：实现登录 API
第三步：实现登录页面
第四步：验证是否满足 SPEC
```

#### 3. 定期验证

```markdown
当前完成：用户注册功能
请对照 SPEC.md 2.2 节验收标准逐项检查。
```

### 处理 AI 失误

#### AI 添加额外功能

```markdown
请停止。你刚才添加的 [功能名称] 不在 SPEC.md 中。
请确认 SPEC 中的功能列表，只实现那些功能。
如确实需要添加，请先更新 SPEC。
```

#### AI 忽略约束

```markdown
请重新阅读 SPEC.md [具体章节]。
你的实现不符合 [具体要求]。
请修改代码以满足 SPEC 规格。
```

## 团队协作最佳实践

### SPEC 评审

#### 评审要点

1. **完整性检查**
   - [ ] 所有需求都有对应规格
   - [ ] 所有规格都有验收标准
   - [ ] 边界情况已考虑

2. **一致性检查**
   - [ ] 功能间无冲突
   - [ ] 约束条件合理
   - [ ] 术语统一

3. **可行性检查**
   - [ ] 技术上可实现
   - [ ] 资源充足
   - [ ] 时间合理

### 代码审查

#### 审查清单

- [ ] 代码是否符合 SPEC 规格
- [ ] 验收标准是否全部满足
- [ ] 是否有未记录的变更
- [ ] 代码风格是否统一
- [ ] 是否有适当的测试

### Git 提交规范

```
feat: 实现登录功能 (F001)
├── 实现内容：用户名密码登录
├── 符合 SPEC：2.1 功能 1
└── 验收标准：全部通过

fix: 修复会话超时问题
├── 问题：用户操作中突然登出
├── 原因：token 过期时间过短
└── 修复：延长 token 有效期至 7 天
```

## 常见问题与解决方案

### Q1: 需求频繁变更怎么办？

**问题**：业务方经常变更需求，导致 SPEC 跟不上。

**解决**：
1. 建立变更流程，需书面申请
2. 评估变更影响后再实施
3. 记录变更次数和时间
4. 与业务方约定变更成本

### Q2: AI 不遵守 SPEC 约束

**问题**：AI 总是添加额外功能或偏离规格。

**解决**：
1. 在提示词中明确列出"禁止项"
2. 频繁让 AI 对照 SPEC 自检
3. 使用更严格的约束语言
4. 必要时限制 AI 的操作范围

### Q3: SPEC 与代码不一致

**问题**：代码更新后忘记同步 SPEC。

**解决**：
1. 强制要求代码提交前检查 SPEC
2. 使用 CI 验证 SPEC 变更声明
3. 定期（每周）进行 SPEC 与代码对比
4. 将 SPEC 更新纳入 Definition of Done

### Q4: SPEC 过于详细导致僵化

**问题**：SPEC 写得太死，无法适应变化。

**解决**：
1. 区分核心规格和实现细节
2. 核心规格严格遵守
3. 实现细节可灵活调整
4. 使用"宜粗宜细"的原则

## SDD 度量指标

### 项目健康度

| 指标 | 计算方式 | 健康值 |
|------|----------|--------|
| SPEC 覆盖率 | 有 SPEC 的功能 / 总功能 | > 90% |
| 规格一致性 | 符合 SPEC 的代码 / 总代码 | > 95% |
| 变更同步率 | 同步更新的变更 / 总变更 | > 90% |
| 验收通过率 | 一次通过的功能 / 总功能 | > 80% |

### 效率指标

| 指标 | 计算方式 | 目标值 |
|------|----------|--------|
| 返工率 | 返工工时 / 总工时 | < 20% |
| 需求澄清次数 | 口头澄清次数 | < 3 次/功能 |
| 验收迭代次数 | 平均迭代次数 | < 2 次 |

## 总结

SDD 成功的关键：

1. **好的 SPEC** - 清晰、可测试、不过不失
2. **严格执行** - 规范是规范，不是参考
3. **持续同步** - 代码和 SPEC 保持一致
4. **善用 AI** - AI 是工具，不是决策者
5. **团队共识** - 大家都认可并遵循

记住：**SPEC 是契约，不是建议。**
