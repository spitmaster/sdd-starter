# olddocs 吸收指南

> AI 自动吸收历史文档的标准流程

---

## 📋 任务说明

当项目存在 `olddocs/` 目录时，AI 应自动执行以下吸收流程，将历史文档中的有价值信息整合到 `docs/` 体系。

---

## 🔄 吸收流程

### 第一步：扫描目录

```
1. 检查项目根目录是否存在 olddocs/
2. 如果存在，递归扫描所有文件和子目录
3. 建立文档清单
```

### 第二步：阅读分析

按优先级阅读各类文档：

| 优先级 | 文档类型 | 吸收目的 |
|--------|----------|---------|
| P0 | 需求文档/ | 提取核心功能需求 |
| P1 | 技术设计/ | 提取架构决策 |
| P2 | API文档/ | 提取接口规范 |
| P3 | 项目笔记.md | 提取上下文信息 |
| P4 | 术语表.md | 提取专业术语 |

### 第三步：智能整合

根据文档类型，将内容整合到对应位置：

#### 3.1 需求文档 → docs/01-使用说明.md
- 提取核心功能列表
- 提取用户角色定义
- 提取优先级信息
- 融入「功能概览」章节

#### 3.2 技术设计 → docs/02-SDD方法论.md
- 提取架构图和说明
- 提取技术选型决策
- 提取数据库设计
- 融入「技术架构」章节

#### 3.3 API文档 → docs/templates/API-Template.md
- 提取接口列表
- 提取请求/响应格式
- 提取错误码定义
- 创建或更新 API 章节

#### 3.4 项目笔记 → 相关章节
- 提取关键决策记录 (ADR)
- 融入项目背景说明
- 补充上下文信息

#### 3.5 术语表 → docs/05-最佳实践.md
- 提取术语定义
- 创建术语表章节
- 补充业务解释

### 第四步：生成摘要

创建 `docs/olddocs-摘要.md`，包含：

```markdown
# olddocs 吸收摘要

## 吸收时间
{时间戳}

## 原文档清单
| 文件 | 类型 | 关键内容 |
|------|------|---------|
| olddocs/需求文档/PRD.md | PRD | 3个核心功能 |

## 吸收结果
### 新增章节
- 「3.1 核心功能需求」- 来自 PRD

### 更新章节
- 「2.3 技术架构」- 新增架构图

### 保留未整合
- {说明原因}

## 建议确认
- [ ] 核心功能列表是否符合预期
- [ ] 技术架构描述是否准确
- [ ] 术语解释是否正确
```

### 第五步：用户确认

```
吸收完成！已生成摘要供你确认。
请检查 docs/olddocs-摘要.md 中的内容是否准确。
如需调整，可直接修改对应文档。
```

---

## ⚠️ 注意事项

1. **不覆盖**：不删除原始文档，只整合内容
2. **不冲突**：遇到冲突时，优先保留新结构，附加说明
3. **可追溯**：保留 olddocs/ 便于后续追溯
4. **需确认**：生成摘要后等待用户确认

---

## 📝 吸收提示词模板

```
请执行 olddocs 吸收流程：

1. 扫描 olddocs/ 目录结构
2. 阅读所有文档内容
3. 按类型智能整合到 docs/ 体系
4. 生成吸收摘要到 docs/olddocs-摘要.md
5. 告知用户吸收结果并等待确认
```
