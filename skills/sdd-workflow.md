# sdd-workflow - SDD 工作流程 Skill

> 提供 SDD (Specification-Driven Development) 工作流程的逐步指导

---

## Skill 信息

- **名称**: sdd-workflow
- **描述**: 指导用户按照 SDD 规范进行开发
- **触发词**: `SDD 工作流程`, `开始 SDD 开发`, `SPEC 驱动开发`

---

## 工作流程概述

SDD 工作流程分为 5 个阶段：

```
1. 需求收集 → requirements.md (可选)
2. SPEC 编写 → SPEC.md (必须)
3. 开发实现 → 代码实现
4. 自测验证 → 对照验收标准
5. 验收确认 → 交付
```

---

## 阶段 1: 需求收集

### 目标

将模糊的想法转化为结构化的需求文档。

### 步骤

1. **收集原始需求**
   - 与用户沟通，了解他们想要什么
   - 识别功能点和非功能需求
   - 记录约束条件和边界

2. **输出 requirements.md**（可选）
   ```
   requirements.md
   ├── 背景和动机
   ├── 用户故事
   ├── 功能需求
   ├── 非功能需求
   └── 边界和约束
   ```

### 示例触发

```
用户：「我想做个微信小程序，记录力量训练」
AI：
  1. 识别关键功能：训练记录、动作库、统计数据
  2. 识别非功能需求：离线使用、数据同步
  3. 生成 requirements.md
```

---

## 阶段 2: SPEC 编写

### 目标

编写 `SPEC.md`，作为开发的唯一真相源。

### SPEC.md 结构

```markdown
# SPEC.md

## 功能概述
[简短描述功能]

## 功能需求
### F1: [功能名称]
- 描述：[功能描述]
- 优先级：高/中/低
- 验收标准：
  - AC1: [可验证的验收标准]
  - AC2: ...

## 非功能需求
- 性能：[要求]
- 安全：[要求]
- 可用性：[要求]

## 技术设计
- 架构：[描述]
- 数据模型：[描述]
- API 设计：[描述]

## 验收标准总览
- [ ] F1-AC1
- [ ] F1-AC2
- [ ] ...
```

### 步骤

1. **基于 requirements.md 编写 SPEC**
2. **与用户确认验收标准**
3. **获得用户批准后开始开发**

### 重要原则

- **验收标准必须可验证**
- **SPEC 是唯一真相源** - 后续所有工作基于 SPEC
- **变更必须更新 SPEC** - 需求变更时先更新 SPEC

---

## 阶段 3: 开发实现

### 目标

按照 SPEC.md 进行开发，频繁验证。

### 步骤

1. **读取 SPEC**
   - 理解功能需求和验收标准
   - 制定开发计划（可选，复杂功能）

2. **实现功能**
   - 小步迭代，频繁验证
   - 对照验收标准检查
   - 及时修复问题

3. **更新文档**
   - 如有设计变更，先更新 SPEC
   - 更新代码注释（如有必要）

### 开发原则

- **先 SPEC，后代码**
- **不要偏离 SPEC** - 如需变更，先更新 SPEC
- **频繁验证** - 每完成一个小功能点就验证

---

## 阶段 4: 自测验证

### 目标

对照 SPEC 中的验收标准，逐一验证功能。

### 步骤

1. **创建测试清单**
   ```
   - [ ] F1-AC1: 验证 [具体场景]
   - [ ] F1-AC2: 验证 [具体场景]
   - [ ] ...
   ```

2. **执行测试**
   - 手动测试或自动化测试
   - 记录测试结果
   - 修复发现的问题

3. **回归测试**
   - 确保修改不破坏已有功能
   - 运行相关测试套件

### 测试原则

- **验收标准驱动** - 每个 AC 都必须验证
- **真实场景** - 在真实或模拟环境中测试
- **边界条件** - 测试异常和边界情况

---

## 阶段 5: 验收确认

### 目标

确认满足 SPEC 中的所有验收标准，准备交付。

### 步骤

1. **最终验证**
   - 逐一检查验收标准
   - 确保所有 AC 都通过

2. **代码审查**（可选）
   - 对照 SPEC 审查代码
   - 检查代码质量

3. **交付**
   - Commit 代码
   - 更新文档（如 roadmap.md、milestones.md）
   - 通知用户验收完成

### Commit 规范

- **完成完整功能后再 commit** - 不要一有改动就 commit
- **Commit message 要清晰** - 使用 `feat:`, `fix:`, `docs:` 等前缀
- **测试通过后再 commit** - 确保功能正常

## 常见场景

### 场景 A：有完整的 PRD 或需求文档

```
1. 读取 PRD/需求文档
2. 生成 SPEC.md
3. 确认后开始开发
```

### 场景 B：只有模糊想法

```
1. 与用户沟通，收集需求
2. 生成 requirements.md（可选）
3. 生成 SPEC.md
4. 确认后开始开发
```

### 场景 C：已有项目，继续开发

```
1. 读取现有 SPEC.md
2. 了解当前进度（读取 todo.md）
3. 继续开发或更新 SPEC
```

### 场景 D：有老项目文档需要迁移

```
1. 将老文档放到 olddocs/
2. 执行吸收流程，更新 docs/
3. 生成或更新 SPEC.md
```

## 最佳实践

1. **SPEC 先行** - 任何代码修改前先确认 SPEC
2. **验收标准驱动** - 不满足 AC 不交付
3. **文档同步** - 代码变更时同步更新 SPEC
4. **小步迭代** - 频繁验证，及时发现问题
5. **一次性 commit** - 完成完整功能后再 commit

## 常见问题

### Q: SPEC 写得不详细怎么办？

A: 先按当前理解编写，开发过程中发现不清楚的地方，立即补充 SPEC。

### Q: 开发中需求变更了怎么办？

A: **先更新 SPEC**，再继续开发。保持 SPEC 与代码同步。

### Q: 如何知道功能是否完成？

A: 对照 SPEC 中的验收标准总览，确保所有 AC 都验证通过。

### Q: Commit 的频率如何把握？

A: **完成一个完整功能后再 commit**。不要一有改动就 commit。
